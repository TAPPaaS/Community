# euro-office — TAPPaaS Module

Collaborative document editing platform based on [Euro-Office DocumentServer](https://github.com/Euro-Office/DocumentServer), an OnlyOffice-compatible document server. Provides real-time co-authoring for Word, Excel, and PowerPoint-format documents directly in the browser.

**Status:** Testing — not intended for production use.

**VMID:** 343 · **Node:** tappaas1 · **Zone:** srv · **Disk:** 32 GB on tanka1
**Public URL:** https://eu-office.example.com · **Internal URL:** http://euro-office.srv.internal (port 80)

---

## What Gets Installed

- Browser-based editing of `.docx`, `.xlsx`, and `.pptx` documents
- Real-time collaborative editing (multiple users simultaneously)
- Built-in test application at `/example/` for verifying the installation
- OnlyOffice API-compatible — integrates with Nextcloud, Seafile, and custom apps

## Architecture

A single Podman container bundles all required services internally:

| Component | Role |
|-----------|------|
| Nginx | HTTP reverse proxy, serves the editor UI (port 80) |
| DocService (Node.js) | Document co-authoring engine |
| FileConverter (C++) | Converts between document formats |
| PostgreSQL | Document session and metadata storage |
| Redis | Session state cache |
| RabbitMQ | Internal message bus |
| Example app | Built-in test interface at `/example/` |

Only port 80 is exposed to the host network. All other ports are internal to the container.

## Security Notes

- JWT authentication is configured automatically. A 64-char hex `JWT_SECRET` is auto-generated on first boot and stored in `/etc/secrets/euro-office.env` on the VM. The same secret is pushed to Nextcloud's OnlyOffice connector during install. To re-sync after a manual rotation, run `update-jwt.sh`.
- SSH password authentication is disabled; only key-based login is permitted.
- The document data volume (`/var/lib/euro-office/data`) is root-owned with mode `0700`.

## Secrets

| File | Contents |
|------|----------|
| `/home/tappaas/secrets/euro-office.env` | `TAPPAAS_PASSWORD` (tappaas user password for cloud-init) |
| `/etc/secrets/euro-office.env` (inside VM) | `JWT_SECRET` — auto-generated on first boot via 64-char hex |

The `JWT_SECRET` is generated by a systemd oneshot on first VM boot and is never written to the TAPPaaS host.

## Backups

Documents are backed up daily at 02:00 by stopping the container, archiving `/var/lib/euro-office/data`, and restarting. Backups older than 30 days are automatically deleted. Archives are stored at `/var/backup/euro-office/` on the VM.

## Debugging

### Container status

```bash
# Check if the container is running
ssh tappaas@euro-office.srv.internal "sudo podman ps"

# Restart the container
ssh tappaas@euro-office.srv.internal "sudo systemctl restart podman-euro-office"
```

### Log access

All logs are inside the container. Get the container ID first:

```bash
CID=$(ssh tappaas@euro-office.srv.internal "sudo podman ps -q")
```

| Log | Command |
|-----|---------|
| DocService (document engine) | `ssh tappaas@euro-office.srv.internal "sudo podman exec $CID tail -100 /var/log/onlyoffice/documentserver/docservice/out.log"` |
| FileConverter errors | `ssh tappaas@euro-office.srv.internal "sudo podman exec $CID tail -50 /var/log/onlyoffice/documentserver/converter/err.log"` |
| Nginx errors | `ssh tappaas@euro-office.srv.internal "sudo podman exec $CID tail -50 /var/log/onlyoffice/documentserver/nginx.error.log"` |

### Stale document sessions

If a document reports "cannot be accessed right now" after a crash or interrupted edit, a stale session may be blocking it. Clear it inside the container's PostgreSQL:

```bash
CID=$(ssh tappaas@euro-office.srv.internal "sudo podman ps -q")
# Find stale sessions (status=1 means currently editing)
ssh tappaas@euro-office.srv.internal "sudo podman exec $CID su -s /bin/sh postgres -c \"psql -d onlyoffice -c 'SELECT id, status FROM task_result WHERE status=1'\""
# Delete by document key (replace KEY with the key shown)
ssh tappaas@euro-office.srv.internal "sudo podman exec $CID su -s /bin/sh postgres -c \"psql -d onlyoffice -c \\\"DELETE FROM task_result WHERE id LIKE 'KEY%'\\\"\""
```

### JWT secret

The JWT secret is auto-generated on first boot and never needs to change during normal operation. It is only relevant if you need to rotate it manually.

**View the current secret:**
```bash
ssh tappaas@euro-office.srv.internal "sudo cat /etc/secrets/euro-office.env"
```

**Rotate the JWT secret** (only needed if the secret is compromised or you want to force rotation):

1. Edit the secret on the VM:
   ```bash
   ssh tappaas@euro-office.srv.internal \
     "echo JWT_SECRET=\$(openssl rand -hex 32) | sudo tee /etc/secrets/euro-office.env"
   ```
2. Restart the container so it picks up the new value:
   ```bash
   ssh tappaas@euro-office.srv.internal "sudo systemctl restart podman-euro-office"
   ```
3. Push the new secret to the Nextcloud OnlyOffice connector:
   ```bash
   cd /home/tappaas/TAPPaaS/src/apps/euro-office
   ./update-jwt.sh
   ```

`update-jwt.sh` reads the current `JWT_SECRET` from the VM and writes it — along with the connector URLs — to Nextcloud's OnlyOffice app config. Normal updates (`update-module euro-office`) do **not** run this script.
